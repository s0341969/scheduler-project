# 排程系統導入 ML/APS 的落地藍圖

## 文件定位

- 本文件是評估版藍圖。
- 目的：評估既有排程系統是否適合導入 `ML` 與 `APS`，以及應採取的最小風險落地順序。
- 本文件不代表立即改版。
- 本文件不要求變更既有 `dbo.產生ORDE3剩餘製程` 的正式執行流程。

## 目前系統判讀

依目前專案與既有 SQL 邏輯判斷，現況屬於：

- 以 SQL Stored Procedure 為核心的排程與剩餘製程計算系統。
- 大量商業規則直接內嵌在 SQL 中。
- 系統強項是規則完整、可直接落地。
- 系統弱項是規則多、維護成本高、預估能力不足、最佳化空間有限。

目前這套系統比較像：

- `Rule-based scheduling engine`

而不是：

- `Data-driven predictive scheduling engine`

這代表它非常適合導入 `ML` 做預測補強，也適合導入 `APS` 做排程決策輔助，但不適合一開始就直接改成黑盒 AI 自動排程。

## 是否適合導入

結論：適合，但應採用漸進式導入。

適合的原因：

- 已經有大量歷史排程、報工、機台、交期、製卡資料，可作為訓練基礎。
- 既有系統已經有穩定規則，可以作為第一版基準線。
- 目前系統最缺的不是規則，而是：
  - 預測實際工時
  - 預測延誤風險
  - 預測等待時間
  - 多方案比較能力

不適合直接一步到位的原因：

- 商規過多且部分屬於隱性現場知識。
- 許多規則目前仍靠人工經驗值修正。
- 若直接導入黑盒排程模型，排錯時難以追原因。
- 現場通常需要可解釋結果，而不是只有最佳分數。

## 建議導入原則

- 先預測，後決策。
- 先輔助，後取代。
- 先旁路，後核心。
- 先離線驗證，後小範圍上線。

翻成實務語言：

- 第一階段先讓模型提供參考值，不直接改正式排程結果。
- 第二階段再讓 APS 用模型結果做排序建議。
- 第三階段才考慮局部接管某些製程或機台族群。

## 可以先導入的能力

### 1. 製程工時預測

用途：

- 取代部分固定工時、估算工時、人工經驗係數。
- 讓 `剩餘工時`、`可用工時`、`CUS工時` 等欄位更接近現場。

建議預測目標：

- 單製卡單製程實際工時
- 單製卡單製程等待工時
- 外包實際週轉時間

可用特徵：

- 圖號
- 製卡
- 製程代碼
- 製程群組
- 機台
- 客戶
- 線別
- 製卡階層
- 是否首件
- 是否外包
- 歷史報工次數
- 歷史平均工時
- 交期壓力

### 2. 延誤風險預測

用途：

- 先找出高風險製卡，再反推優先順序。
- 讓目前純規則式排序增加風險導向能力。

建議預測目標：

- 製卡是否將逾期
- 訂單是否將逾期
- 某製程是否會卡站

輸出建議：

- `延誤風險分數`
- `預測逾期天數`
- `風險等級`

### 3. 機台/站點等待時間預測

用途：

- 改善目前排程對實際壅塞的反映能力。
- 補足規則無法即時反映現場塞車的缺口。

建議預測目標：

- 某機台未來 4 小時等待量
- 某製程平均排隊時間
- 某線別即將擁塞機率

### 4. 異常偵測

用途：

- 提前發現不合理排程結果。
- 提前標示異常工時、異常跳站、異常報工模式。

建議目標：

- 工時異常
- 製程流異常
- 報工時間異常
- 外包週轉異常

## 不建議一開始就做的能力

### 1. 直接用強化學習主導全排程

原因：

- 狀態空間太大。
- 現有資料多為歷史結果，不足以直接支撐線上探索。
- 現場不可接受高試錯成本。

### 2. 直接用單一黑盒模型取代 SQL 商規

原因：

- 商規很多且高度情境化。
- 可解釋性不足。
- 回歸驗證成本高。

### 3. 一次重寫整個排程核心

原因：

- 風險過高。
- 業務影響過大。
- 無法快速驗證收益。

## 建議的系統架構

建議保留現有 SQL 核心，外加一層 `ML/APS` 服務。

建議架構如下：

- `ERP / MES / 報工資料`
- `SQL 資料整理層`
- `Feature Store / 訓練資料集`
- `ML 預測服務`
- `APS 決策服務`
- `結果回寫 SQL`
- `現有查詢 / 看板 / 排程輸出`

其中各層責任建議如下：

- SQL：
  - 負責資料清洗
  - 負責製卡、製程、報工、機台主鍵整併
  - 負責結果回寫
- ML 服務：
  - 負責工時預測
  - 負責延誤風險預測
  - 負責等待時間預測
- APS 服務：
  - 負責多目標排序
  - 負責方案評分
  - 負責排程模擬

## 對既有 SQL 系統的接法

不直接改寫正式 SP，建議先用旁路方式接入。

### 方案 A：只讀評估模式

- 現有 SP 不變。
- 每次 SP 輸出後，額外產出一份 `ML/APS 評估結果`。
- 比較：
  - 原排序
  - 模型建議排序
  - 延誤風險差異

適合：

- 先驗證資料品質與模型可用性。

### 方案 B：建議欄位回寫模式

- 現有 SP 不變。
- 新增欄位或旁表，回寫：
  - 預測工時
  - 延誤風險分數
  - 建議優先級
  - 建議機台

適合：

- 讓現場先看建議，不強制採用。

### 方案 C：局部製程接管模式

- 僅針對特定製程群導入 APS 決策。
- 建議優先從：
  - CMM
  - HM
  - 外包
  - 高延誤風險單

適合：

- 小範圍驗證實際效益。

### 第一批資料表現成可用

- 指派時間
- 機台資料
- 客戶資料
- 外包資料

### 第一版訓練資料集建議欄位

- 製卡
- 訂單
- 圖號
- 製程
- 製程群
- 機台
- 線別
- 客戶
- 首件註記
- 外包註記
- 開單日
- 交期
- 在站製程序
- 歷史報工開始時間
- 歷史報工完成時間
- 實際工時

### 第一批資料治理工作

- 主鍵對齊

- 機台代碼標準化
- 客戶群組標準化

## 分階段落地建議

### 階段 0：資料盤點

目標：

- 確認哪些欄位可直接訓練
- 確認歷史資料覆蓋率
- 確認可驗證 KPI

產出：

- 資料字典
- 訓練樣本定義
- KPI 定義

### 階段 1：離線預測 PoC

目標：

- 建立工時預測模型
- 建立延誤風險模型

衡量：

- 工時預測誤差
- 逾期分類準確率
- 高風險單召回率

### 階段 2：建議模式上線

目標：

- 讓現場可以看到建議但不改正式排程

衡量：

- 建議採納率
- 被採納後的延誤改善幅度
- 高風險單提前處理比例

### 階段 3：局部 APS 接管

目標：

- 只在少數製程導入 APS 排序邏輯

衡量：

- 在製品等待時間
- 機台利用率
- 延誤訂單比例

### 階段 4：多目標智慧排程

目標：

- 同時考慮交期、工時、機台利用率、換線成本、外包週轉

衡量：

- 準交率
- 平均在製時間
- 平均換線次數
- 平均急單插單成本

## 評估 KPI

導入 ML/APS 後，建議固定追蹤：

- 準交率
- 平均延誤天數
- 高風險單預警命中率
- 平均製卡等待時間
- 平均製程實際工時誤差
- 機台利用率
- 急單插單影響範圍
- 排程重排次數

## 主要風險

### 1. 資料品質不足

最常見風險：

- 報工時間不一致
- 機台代碼不一致
- 製程代碼歷史沿革太多
- 首件與特殊件標註不完整

### 2. 現場不信任黑盒建議

若沒有解釋欄位，現場通常不會採用。

因此建議輸出：

- 為何風險高
- 為何建議先做
- 為何建議換機台

### 3. 模型與商規衝突

ML 與既有 SQL 規則可能給出不同結論。

因此第一版必須：

- 並行比較
- 不直接覆蓋
- 保留人工最終決策

## 最務實的結論

對你目前這套系統，最可行路線不是：

- 直接把 SQL 排程改成 AI 自動排

而是：

- 用 ML 補預測
- 用 APS 補決策
- 用現有 SQL 保持可執行與可追溯

也就是：

- `SQL 保核心`
- `ML 做預測`
- `APS 做建議`
- `現場逐步採用`

## 建議下一步

若之後要往下走，建議順序如下：

1. 先整理一份訓練資料欄位清單
2. 先做工時預測 PoC
3. 再做延誤風險 PoC
4. 再決定是否做局部 APS 排程服務

本文件目前結論僅供評估，不直接進入既有專案正式執行面。
